Claude Code Security: Permissions, Sandboxing & Safe Usage (2026)

By Eric Coste March 5, 2026 10 min read

Understanding the Threat Model

Claude Code runs with your user permissions. Every command it executes, every file it reads, every network request it makes happens as you. A misconfigured Claude Code can read your SSH keys, access your .env files, exfiltrate data, or modify system configuration files.

The primary risk isn't that Claude is malicious — it's that prompt injection through malicious code, packages, or file contents can manipulate Claude's behavior. A poisoned npm package, a crafted README, or a malicious API response could instruct Claude to run destructive commands.

The Permission System

Claude Code's security model has four layers: permission modes, allow/deny rules, sandboxing, and CLAUDE.md instructions. Each layer addresses different risks, and they work together for defense in depth.

Permission modes

Claude Code supports five permission modes: Normal (ask for everything), Auto-edit (auto-approve edits, ask for commands), Auto-approve (approve everything on your allowlist), YOLO (skip all checks — dangerous), and Plan (read-only, no writes or executions).

For most developers, Normal mode with sandboxing enabled gives the best balance of productivity and safety.

Enable Sandboxing

Sandboxing is the single most important security feature in Claude Code. It reduces permission prompts by 84% while actually increasing security. Enable it now:

# Inside Claude Code:
/sandbox

# This opens a menu to choose your mode:
# - Auto-allow: sandbox auto-approves safe commands
# - Regular: still asks permission, but enforces sandbox limits

# On Linux, install prerequisites first:
sudo apt install bubblewrap socat

What sandboxing protects

Filesystem isolation restricts writes to your project directory only. Claude cannot modify ~/.bashrc, /usr/local/bin, or any file outside your working directory. Network isolation routes all outbound connections through a proxy that only allows approved domains. Even if a prompt injection instructs Claude to exfiltrate your SSH keys, the sandbox blocks both the file read and the network request.

Configuring Deny Rules

Deny rules are your strongest defense — they cannot be overridden by any allow rule at any scope level.

// ~/.claude/settings.json (global — applies everywhere)
{
  "permissions": {
    "deny": [
      "Read(**/.env)",
      "Read(**/.env.*)",
      "Read(~/.ssh/**)",
      "Read(**/.credentials*)",
      "Read(**/secrets/**)",
      "Bash(rm -rf*)",
      "Bash(curl*)",
      "WebFetch"
    ]
  }
}

Prompt Injection Defense

Prompt injection occurs when malicious content in a file or API response manipulates Claude's behavior. The permission system is your first defense — even if an injected prompt instructs Claude to run rm -rf /, Normal mode requires your confirmation.

Hooks provide a second defense layer. A PreToolUse hook that blocks specific commands fires at every execution point, regardless of what's in the conversation context:

// In your settings.json hooks configuration
{
  "hooks": {
    "PreToolUse": [{
      "matcher": "Bash",
      "command": "if echo \"$TOOL_INPUT\" | grep -qE 'rm -rf|curl.*\|.*bash|wget.*\|'; then echo 'BLOCKED: Dangerous command pattern detected'; exit 1; fi"
    }]
  }
}

Team Security with Managed Settings

For teams, use a managed-settings.json file that enforces security policies organization-wide. This file takes highest precedence and cannot be overridden by individual developers.

Distribute it through your deployment pipeline, CI/CD system, or MDM solution. Include mandatory deny rules, required sandbox mode, and approved MCP server lists.

Security Checklist

Before using Claude Code on any project with sensitive data, verify these settings: sandboxing is enabled (/sandbox), deny rules block .env and credential files, network isolation prevents unauthorized outbound connections, MCP servers are explicitly approved (not auto-accepted), and hooks block destructive command patterns.

Frequently Asked Questions

Is Claude Code safe to use?

Claude Code is safe when properly configured. Enable sandboxing (/sandbox), use deny rules for sensitive directories, and never use --dangerously-skip-permissions on a machine with sensitive data. The multi-layered permission system protects against both accidental and malicious actions.

What is sandboxing in Claude Code?

Sandboxing uses OS-level primitives (Seatbelt on macOS, bubblewrap on Linux) to isolate Claude Code's processes. It restricts filesystem access to your project directory and network access to approved domains, preventing data exfiltration even if a prompt injection succeeds.

Should I use --dangerously-skip-permissions?

Only in completely isolated environments like Docker containers with no internet access and no sensitive data. Never on your development machine. The sandbox feature provides a much safer alternative that reduces permission prompts by 84% without removing all safety guardrails.

How do I protect my SSH keys and .env files?

Add deny rules in your settings.json: permissions.deny should include 'Read(**/.env)', 'Read(~/.ssh/**)', and 'Read(**/.credentials*)'. Sandboxing also blocks access to directories outside your project by default.
